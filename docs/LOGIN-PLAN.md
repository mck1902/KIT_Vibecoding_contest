# 로그인 기능 추가 분석 및 구현 플랜

## 개요

현재 프로젝트에는 실제 로그인 기능이 없습니다.  
존재하는 것은 역할 선택 후 학생 화면 또는 학부모 화면으로 이동하는 목업 수준의 진입 흐름입니다.

문서 기준으로도 현재 로그인은 정식 인증이 아니라 데모용 역할 선택에 가깝습니다.

## 현재 상태 분석

### 1. 프론트엔드

- `frontend/src/pages/Login.jsx`
  - 이메일/비밀번호 입력 없이 학생 / 학부모 버튼 클릭만으로 페이지 이동
- `frontend/src/App.jsx`
  - `/student`, `/parent` 라우트가 보호되지 않음
- `frontend/src/pages/StudentDashboard.jsx`
  - 세션 생성 시 `demo-student-001` 하드코딩 사용
- `frontend/src/pages/ParentDashboard.jsx`
  - 로그인 사용자 기준 조회가 아니라 전달 상태값과 목업에 의존

### 2. 백엔드

- `backend/src/index.js`
  - 인증 관련 라우트 없음
- `backend/src/models/Session.js`
  - 세션 모델만 존재
- 사용자 계정, 비밀번호, 역할, 권한 검증 관련 모델과 미들웨어가 없음

### 3. docs 기준 해석

- `docs/SPEC-FRONTEND.md` 에서는 로그인 요구사항이 실질적으로
  - "학생으로 입장"
  - "학부모로 입장"
  수준으로만 정의되어 있음
- 즉 현재 문서는 정식 인증 시스템을 요구하지 않음

## 결론

냉정하게 보면, "로그인 기능 추가"는 현재 문서 범위의 보강이 아니라 기능 확장입니다.

따라서 구현은 다음 중 하나로 명확히 정해야 합니다.

1. 데모용 로그인 유지
2. 최소 인증 기능 추가
3. 정식 회원 시스템까지 확장

현재 코드베이스 상태를 기준으로는 `최소 인증 기능 추가`가 가장 현실적입니다.

## 권장 범위

### MVP 권장안

- 이메일/비밀번호 로그인
- 학생 / 학부모 2개 역할
- JWT 기반 로그인 상태 유지
- 보호 라우트 적용
- 로그인한 사용자 기준으로 세션/리포트 접근 제한

### 지금 단계에서 비권장

- OAuth 로그인
- 회원가입/비밀번호 재설정까지 전체 구현
- 다중 자녀/복잡한 권한 체계 동시 도입

이유:

- 현재 사용자 도메인이 비어 있음
- 세션/리포트 기능보다 인증 설계 비용이 더 커질 가능성이 높음

## 구현 플랜

### 1. 인증 범위 확정

- 데모용인지 실제 인증인지 먼저 확정
- 권장안은 JWT 기반 MVP 로그인

정해야 할 항목:

- 학생과 학부모 모두 이메일 로그인인지
- 회원가입이 필요한지
- 학부모가 여러 자녀를 가져야 하는지

위 항목은 현재 docs만으로는 확정할 수 없습니다.

### 2. 백엔드 사용자 도메인 추가

추가 대상:

- `User` 모델

권장 필드:

- `email`
- `passwordHash`
- `role` (`student` | `parent`)
- `name`
- `studentId`
- `parentId`

주의:

- 현재 `Session.studentId` 는 문자열 기반이므로 로그인 사용자와의 매핑 규칙을 먼저 정리해야 함

### 3. 인증 API 추가

필수 API:

- `POST /api/auth/login`
- `GET /api/auth/me`
- `POST /api/auth/logout` 또는 프론트 토큰 삭제 방식

권장 기술:

- `bcrypt` 로 비밀번호 해시
- `jsonwebtoken` 으로 JWT 발급
- `Authorization: Bearer <token>` 방식 사용

### 4. 권한 미들웨어 추가

필수 미들웨어:

- `requireAuth`
- `requireRole('student')`
- `requireRole('parent')`

권한 정책:

- 학생은 자기 세션만 생성/조회 가능
- 학부모는 연결된 자녀 세션만 조회 가능

### 5. 프론트 인증 상태 관리 추가

추가 대상:

- `AuthContext` 또는 전역 상태 관리 레이어

필요 기능:

- 로그인 성공 시 토큰 저장
- 앱 시작 시 `/api/auth/me` 호출
- 로그아웃 처리
- 인증 만료 시 로그인 화면 복귀

### 6. 로그인 페이지 실제 폼으로 교체

현재:

- 역할 선택 목업

변경:

- 이메일 입력
- 비밀번호 입력
- 로그인 버튼
- 데모 계정 안내 문구

로그인 성공 후:

- 학생이면 `/student`
- 학부모면 `/parent`

### 7. 보호 라우트 적용

보호 대상:

- `/student`
- `/parent`
- `/student/report/:sessionId`

정책:

- 비로그인 시 `/login` 리다이렉트
- 역할이 다르면 접근 차단 또는 적절한 홈으로 이동

### 8. 기존 하드코딩 제거

수정 필요:

- `frontend/src/pages/StudentDashboard.jsx`
  - `demo-student-001` 제거
- `frontend/src/pages/ParentDashboard.jsx`
  - 로그인 사용자 기준 데이터 조회로 변경

### 9. 시드 데이터 추가

데모용 최소 계정 생성:

- 학생 계정 1개
- 학부모 계정 1개
- 학생-학부모 연결 관계 1세트

목적:

- 대회 데모 시 즉시 로그인 가능하도록 준비

## 권장 구현 순서

1. `User` 모델 및 인증 라우트 추가
2. JWT/권한 미들웨어 추가
3. 데모 계정 시드 데이터 생성
4. 프론트 인증 컨텍스트 및 보호 라우트 추가
5. 로그인 페이지 폼 전환
6. 학생/학부모 대시보드에 로그인 사용자 연결
7. 세션/리포트 API 권한 검사 적용
8. 로그아웃 및 예외 처리 정리

## 리스크

### 1. 가장 큰 리스크

로그인 자체보다 사용자-학생-학부모 관계 모델링이 더 큽니다.

### 2. 구조적 리스크

- 현재 백엔드에 사용자 개념이 없음
- 현재 세션은 `studentId` 문자열만 기준으로 동작
- 학부모 데이터 접근 규칙이 아직 정식 설계되지 않음

### 3. 범위 리스크

회원가입, 다중 자녀, 계정 복구까지 같이 넣으면 일정 대비 과해질 가능성이 높음

## 최종 판단

현재 프로젝트에 가장 적합한 방향은 다음입니다.

- 정식 회원 시스템 전체 구축: 비추천
- 데모용 역할 선택 유지: 기능 추가 목적에는 부족
- JWT 기반 MVP 로그인 추가: 가장 현실적

즉, 이 프로젝트의 로그인 기능은 `정식 서비스 수준 인증`이 아니라  
`데모 가능한 최소 인증 + 역할 기반 접근 제어`로 구현하는 것이 가장 타당합니다.
